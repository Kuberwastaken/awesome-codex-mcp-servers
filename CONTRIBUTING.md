# Contributing

Thanks for helping keep this list awesome. A curated list is only as good as its bar for entry, so please read this before opening a pull request. PRs that don't follow these guidelines may be closed without review — nothing personal, it just keeps the list trustworthy.

## What belongs here

An MCP server earns a spot if it is:

- **Real and usable** — it implements the Model Context Protocol and works with the Codex CLI, the IDE extension, or Codex cloud today. If it only works with one specific non-Codex client, it probably belongs on that client's list instead.
- **Maintained** — meaningful activity in roughly the last 12 months. Archived or abandoned projects are removed; the only exception is the small set of official MCP `reference`/`archived` servers people still rely on, which are tagged as such.
- **Documented** — a working link and enough of a README that someone can actually install and run it.
- **Past the launch-day stage** — it should be more than an empty repo or a screenshot. A little maturity, some real usage, or clear utility.
- **Not a duplicate** — search the list first. One canonical entry per project. If two servers do the same thing, the more maintained or more widely used one wins.

Low-effort wrappers, link farms, broken links, and pure marketing pages will be declined.

## Entry format

Add your entry as a single line, in the correct category, in this exact shape:

```
- [Name](https://link) - What it does, in one plain sentence. `Lang` `runs` `source`
```

Rules:

- **Name** links to the project's repository or homepage — the canonical source, not a directory listing or a blog post.
- **Separator** is a space, a hyphen, and a space (` - `). Not an en-dash or em-dash; some editors auto-correct the hyphen, so double-check.
- **Description** starts with a capital letter and ends with a period. Keep it to **one objective sentence** — say what it does, not that it's "the best", "blazing-fast", "powerful", or "the only". Skip filler like "open-source", "free", "MCP server", or "for Codex"; the list already implies all of that.
- **Tags** come after the period, as backtick chips, in this order:
  - **Language**: `TS` `Py` `Go` `Rust` `C#` `Java` `JS` `Ruby` (add another if your stack isn't listed).
  - **Runs**: `local`, `remote`, or `local/remote`.
  - **Source**: `reference` (official MCP project server), `official` (maintained by the product's own vendor), or `archived` (archived reference server). Leave this off for community projects — community is the default.

Example:

```
- [Grafana](https://github.com/grafana/mcp-grafana) - Access dashboards, datasources, alerts, and incidents. `Go` `local/remote` `official`
```

## Ordering and categories

- Put the entry in the single best-fit category. If it spans two, pick the primary use case.
- Ordering within a category is by relevance and grouping, not strict alphabetical — place your entry where a reader would expect it, near similar tools.
- Propose a **new category** only if several entries would live in it. Say why in the PR description.

## One suggestion per pull request

- **One project per PR.** Separate suggestions belong in separate PRs so they can be accepted or declined independently, and the git history stays clean.
- Title the PR `Add <Project Name>` (or `Fix ...` / `Remove ...` for maintenance).

## Disclose self-promotion

You're welcome to submit your own project — but **say so in the PR description** ("I maintain this"). Self-submissions are held to the same bar as everything else, and to a slightly higher standard on objectivity in the description.

## No AI-generated submissions

Don't open PRs whose description or entry was generated by an LLM without your review. Curate with your own judgment; that's the whole point of an awesome list.

## Before you open the PR

- [ ] The entry is in the right category and near similar tools.
- [ ] Format, capitalization, ` - ` separator, and trailing period are correct.
- [ ] Language, runs, and source tags are present and accurate.
- [ ] The link works and points to the canonical repo or homepage.
- [ ] The description is one objective sentence.
- [ ] This PR adds, fixes, or removes exactly one project.
- [ ] (Optional but appreciated) `npx awesome-lint` runs clean locally.

Thanks again. Every good entry makes this more useful for the next person.
